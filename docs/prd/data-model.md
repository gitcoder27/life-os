# Data Model

This document defines the backend data model for Life OS MVP. The goal is to keep the schema small enough to build quickly, but structured enough to support scoring, reviews, recurring items, and future module depth without a rewrite.

## Storage decisions

- Primary database: `PostgreSQL`
- Time storage: UTC timestamps, user-facing dates derived with the user's timezone
- Primary key style: UUIDs
- Soft-delete pattern: use `archived_at` or `status` for user-owned records instead of hard deletion where history matters
- Flexible fields: use `JSONB` only for truly variable payloads such as score breakdowns or monthly review outcomes

## High-level relationship map

- One `user` owns almost all records.
- `planning_cycles` group daily, weekly, and monthly planning windows.
- `cycle_priorities` hold the top priorities for a given cycle.
- `goals` can link to priorities and tasks.
- `habits` and `routines` generate daily completion data.
- `health` and `finance` are event-log style tables.
- `reviews` finalize reflection and seed the next cycle.
- `daily_scores` are derived and persisted for fast reads and trend history.
- `notifications` are generated by rules across all modules.

## Core table groups

## 1. Auth and account

### `users`

Purpose: owner account record.

Key fields:

- `id`
- `email` unique
- `password_hash`
- `status` (`active`, `disabled`)
- `onboarded_at` nullable
- `created_at`
- `updated_at`
- `last_login_at`

Notes:

- MVP permits exactly one active owner account, but the schema should not assume only one user forever.

### `sessions`

Purpose: server-managed authenticated sessions.

Key fields:

- `id`
- `user_id`
- `session_token_hash`
- `created_at`
- `last_seen_at`
- `expires_at`
- `ip_address`
- `user_agent`
- `revoked_at`

Constraints:

- index on `session_token_hash`
- index on `user_id, expires_at`

### `audit_events`

Purpose: auth and security audit trail.

Key fields:

- `id`
- `user_id` nullable
- `event_type`
- `event_payload_json`
- `created_at`

Typical events:

- login success
- login failure
- logout
- password reset

### `user_preferences`

Purpose: user-specific settings used across planning, scoring, and review timing.

Key fields:

- `user_id` primary key
- `timezone`
- `currency_code`
- `week_starts_on`
- `daily_review_start_time`
- `daily_review_end_time`
- `daily_water_target_ml`
- `default_score_enabled`
- `created_at`
- `updated_at`

## 2. Planning and execution

### `goals`

Purpose: long-running outcomes across life areas.

Key fields:

- `id`
- `user_id`
- `title`
- `domain` (`health`, `money`, `work_growth`, `home_admin`, `discipline`, `other`)
- `status` (`active`, `paused`, `completed`, `archived`)
- `target_date` nullable
- `notes`
- `created_at`
- `updated_at`

### `planning_cycles`

Purpose: canonical container for day, week, and month planning periods.

Key fields:

- `id`
- `user_id`
- `cycle_type` (`day`, `week`, `month`)
- `cycle_start_date`
- `cycle_end_date`
- `status` (`open`, `review_ready`, `closed`)
- `theme` nullable
- `created_at`
- `updated_at`

Constraints:

- unique on `user_id, cycle_type, cycle_start_date`

### `cycle_priorities`

Purpose: top priorities for a planning cycle.

Key fields:

- `id`
- `planning_cycle_id`
- `slot` (`1`, `2`, `3`)
- `title`
- `status` (`pending`, `completed`, `dropped`)
- `goal_id` nullable
- `source_review_type` nullable
- `completed_at` nullable
- `created_at`
- `updated_at`

Constraints:

- unique on `planning_cycle_id, slot`

### `tasks`

Purpose: lightweight operational tasks and reminders used in Today and Quick Capture.

Key fields:

- `id`
- `user_id`
- `title`
- `notes` nullable
- `status` (`pending`, `completed`, `dropped`)
- `scheduled_for_date` nullable
- `due_at` nullable
- `goal_id` nullable
- `origin_type` (`manual`, `quick_capture`, `carry_forward`, `review_seed`, `recurring`)
- `carried_from_task_id` nullable
- `completed_at` nullable
- `created_at`
- `updated_at`

Notes:

- Tasks are intentionally lightweight and not a separate product module.
- Only tasks scheduled for the current day affect scoring.

- `name`

- `status` (`active`, `archived`)
- `created_at`

Purpose: checklist items inside a routine.

Key fields:

- `id`
- `routine_id`
- `title`
- `sort_order`
- `is_required`

### `routine_item_checkins`

- `id`

## Key indexes and constraints

The backend agent should add these early:

- `planning_cycles(user_id, cycle_type, cycle_start_date)` unique
- `cycle_priorities(planning_cycle_id, slot)` unique
- `tasks(user_id, scheduled_for_date, status)`
- `habit_checkins(habit_id, occurred_on)` unique
- `routine_item_checkins(routine_item_id, occurred_on)` unique
- `workout_days(user_id, date)` unique
- `expenses(user_id, spent_on)`
- `admin_items(user_id, due_on, status)`
- `daily_scores(user_id, finalized_at desc)`
- `notifications(user_id, read_at, dismissed_at, visible_from)`
- `sessions(session_token_hash)` unique

## Derived read models

The backend should keep read aggregation in the application layer for MVP rather than building a large reporting subsystem.

Key read models:

- `HomeOverview`
- `TodayPlan`
- `WeeklyReviewSummary`
- `MonthlyReviewSummary`
- `ScoreBreakdown`

These should be assembled from base tables and cached only if necessary.

## Deletion and history policy

- Keep score, review, expense, and health logs historically intact.
- Archive habits, routines, templates, and categories instead of deleting when possible.
- Task deletion is allowed only for tasks that have no downstream review or scoring references.

## Migration sequencing

Recommended migration order:

1. auth and preferences
2. planning cycles, priorities, goals, tasks
3. habits and routines
4. health logs
5. finance and admin
6. reviews and scores
7. notifications and audit refinements
